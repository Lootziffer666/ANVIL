# Frictionless Design

Frictionless design in ANVIL does not mean aggressive conversion optimization.

It means reducing unnecessary cognitive, operational, emotional, and contextual friction so the user can act with less noise.

## Core Rule

```text
As little interface as possible. As much semantic clarity as necessary.
```

## What Counts As Bad Friction

- repeated manual steps that can be safely removed
- unclear state
- irrelevant dashboards
- premature configuration
- hidden affordances
- unnecessary app switching
- needing to remember what the system already knows
- error messages that describe the system instead of the user's next step

## What Counts As Intentional Friction

- confirmation before destructive writes
- preview before upload/commit/push
- permission boundaries
- safety warnings for irreversible or high-risk actions
- slowdowns that prevent data loss

## Design Bias

Prefer a system that feels calm but operational.

Do not show everything the system knows. Show the one truth the user needs now.
